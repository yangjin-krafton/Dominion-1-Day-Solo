# Dominion 1-Day Solo

도미니언 보드게임을 솔로 플레이 중심으로 재설계한 프로젝트 기획서입니다.  
목표는 **최소 턴/최소 행동으로 목표 승점에 도달하는 최적 루트 탐색**이며, 동일한 게임 코드를 공유해 기록 경쟁이 가능하도록 설계합니다.

## 1. 프로젝트 목적
- 원작 도미니언의 핵심 덱 빌딩 재미를 유지하되, 1인 플레이에 맞춰 규칙과 UI를 단순화한다.
- 모바일 환경에서 빠르게 플레이 가능한 레이아웃/조작 체계를 제공한다.
- 게임 세팅을 코드화해 동일 조건 경쟁(기록 비교, 랭킹)을 가능하게 한다.
- 정적 서빙 환경에서도 동작하며, 개발 환경에서는 데이터 기반 시뮬레이션(인간 vs AI) 확장이 가능하도록 구조화한다.

## 2. 핵심 플레이 컨셉
- 플레이어는 카드 구매/사용을 반복해 목표 승점에 도달한다.
- 게임 성과는 다음 지표로 측정한다.
  - 총 턴 수
  - 총 행동 수(카드 사용, 구매 등)
  - 목표 달성 시점의 덱 효율
- 동일 게임 세팅 기준으로 최저 기록을 만드는 것이 핵심 재미다.

## 3. 솔로 플레이용 규칙 재설계
### 3.1 시장(킹덤) 카드 구성
- 기존: 10장 킹덤 카드 고정
- 변경: **시장 6장 카드 운영**
- 밸런싱 정책:
  - 기준 카드 풀에서 10장 후보를 구성
  - 실제 플레이에는 6장만 노출
  - 플레이/시뮬레이션 로그 기반으로 교체 후보를 순환해 밸런스를 조정

### 3.2 공급 카드 수량 축소
- 솔로 플레이 템포에 맞춰 공급 piles를 축소
- 시장 공급 카드는 **3~5장 범위**에서 설정(카드 타입별 가변)
- 승점/재화 카드도 솔로 템포에 맞게 수량 재조정

### 3.3 상호작용 카드 처리
- 원작의 타 유저 직접 상호작용(공격/방해/반응)은 솔로 기준으로 재설계 대상
- **1차 구현(MVP)에서 제외**
- 추후 확장 시:
  - 가상 상대(NPC/시뮬레이터) 규칙
  - 간접 상호작용(점수 패널티/확률 이벤트) 방식으로 대체 검토

## 4. 승리 조건 및 기록 지표
### 4.1 승리 조건
- 목표 승점 도달 시 승리

### 4.2 기록 지표
- 턴 수
- 전체 행동 카운트
- 구매 횟수
- 카드 사용 횟수
- (선택) 무효 행동/비효율 행동 수

### 4.3 게임 목표 정의
- 단순 승리가 아니라, 동일 세팅에서 **최저 승리 수(최적 기록)**를 찾는 것이 목표

## 5. 게임 코드(세팅 공유) 및 Seed 규칙
### 5.1 핵심 원칙
- 동일 게임 코드는 **완전히 동일한 플레이 환경**을 보장해야 한다.
- 동일 게임 코드에서는 **덱 셔플 순서까지 동일**해야 한다.
- 게임 중 발생하는 모든 랜덤은 단일 seed 기반 PRNG로만 생성한다.

### 5.2 Seed -> 게임 코드 변환
- 게임 생성 시 내부 seed를 먼저 확정한다.
- 이 seed와 룰셋 버전/옵션을 인코딩해 `게임 코드`를 생성한다.
- 사용자는 게임 코드를 공유하고, 수신자는 코드를 디코딩해 동일 seed로 게임을 재현한다.

예시 흐름:
1. `seed` 생성 (또는 지정)
2. `seed + ruleVersion + optionFlags` 직렬화
3. Base32/Base64URL 등으로 인코딩
4. 결과 문자열을 게임 코드로 사용

### 5.3 코드 포함 정보(예시)
- seed (필수)
- 룰셋 버전
- 시장 6장 카드 조합 규칙
- 공급 카드 수량 설정
- 시작 덱/초기 자원 설정
- 목표 승점
- 특수 룰 플래그

### 5.4 랜덤 제어 범위
아래 항목은 모두 seed 기반 난수로 결정한다.
- 게임 시작 시 시장 카드 선택
- 시작 덱 셔플 순서
- 게임 중 셔플 발생 시 덱 재구성 순서
- 이벤트/확률 효과(추후 확장)
- AI 시뮬레이션 내 확률 분기

### 5.5 검증/재현성 요구사항
- 동일 `게임 코드 + 동일 입력`이면 결과 로그가 항상 동일해야 한다.
- 리플레이 모드에서 턴별 상태 해시 검증 가능하도록 설계한다.
- 랭킹 제출 시 게임 코드와 행동 로그를 함께 저장해 재검증 가능하게 한다.

## 6. 랭킹 및 경쟁 구조
- 모든 게임 결과를 기록 대상으로 관리
- 기본 랭킹 기준:
  - 1순위: 목표 달성 턴 수
  - 2순위: 전체 행동 수
  - 3순위: 달성 시간(동률 해소)
- 같은 게임 코드 단위 랭킹과 전체 통합 랭킹을 분리 운영 가능
- 기록 제출 시 `게임 코드(seed 포함)`를 필수값으로 사용

## 6.1 로컬 기록 저장 및 제출
### 로컬 누적 저장
- 플레이 기록은 기본적으로 브라우저 로컬 저장소(IndexedDB/localStorage)에 누적 저장한다.
- 네트워크 없이도 기록 조회/정렬/필터가 가능해야 한다.
- 저장 포맷은 내부 표준 스키마(JSON)로 통일한다.

### 기록 제출 버튼(Export/Submit)
- UI에 `기록 제출` 버튼을 제공한다.
- 제출 방식은 다음 3가지를 지원한다.
  - JSON 파일 내보내기
  - CSV 파일 내보내기
  - 메일 전송용 초안 열기(`mailto:`)
- 기본 사용 흐름:
  1. 기록 파일(JSON/CSV) 다운로드
  2. 메일 초안 열기
  3. 사용자가 파일 첨부 후 직접 발송

### 메일 제출 방식
- 브라우저 보안상 첨부파일 자동 전송은 불가하므로, 다음 흐름으로 처리한다.
  - 사용자가 `기록 제출` 클릭
  - JSON/CSV 파일 다운로드 생성
  - 기본 메일 앱을 `mailto:`로 열고 수신 주소/제목/요약 본문 자동 채움
  - 사용자가 다운로드된 파일을 첨부해 발송
- 수신 주소는 운영용 지정 메일 주소를 사용한다.
- 메일 본문에 아래 내용을 함께 작성하도록 안내한다.
  - 게임 리뷰(재미/난이도/좋았던 점)
  - 개선 요청(불편한 UX, 규칙 개선 의견)
  - 버그 제보(재현 방법 포함)

### GitHub 기록 제출 방식
- 1차 버전에서는 GitHub 자동 업로드를 사용하지 않는다.
- 운영 정책은 `다운로드 + 메일 제출`을 기본 채널로 고정한다.
- 필요 시 운영자가 검수 후 GitHub 저장소에 수동 반영한다.

## 7. 모바일 최적화 UI/UX 방향
- 세로 화면 기준 우선 설계
- 핵심 영역:
  - 상단: 턴/행동/코인/승점
  - 중앙: 손패
  - 하단: 시장/공급 카드 + 구매 액션
- 원탭 중심 조작, 상세 정보는 롱프레스/팝업 제공
- 카드 수 축소 및 우선순위 정렬로 스크롤 최소화

## 8. 기술/운영 아키텍처
### 8.1 배포
- 정적 서빙(예: GitHub Pages) 기준
- 클라이언트 단독 실행 가능한 게임 로직 우선

### 8.2 개발 환경 확장(LLM + 시뮬레이션)
- 개발 PC 환경에서 LLM 연동 가능 구조 추가
- 목적:
  - 카드/세팅 밸런스 검증
  - 자동 플레이 시뮬레이션
  - 인간 vs AI 기록 경쟁 모드 실험
- 주의:
  - 운영 배포는 정적 환경 유지
  - LLM/시뮬레이션은 개발/분석 파이프라인으로 분리

### 8.3 화면 없는 데이터 기반 게임 모델
- UI 없이도 진행 가능한 턴 엔진 필요
- 입력: 게임 코드(= seed 포함 세팅), 플레이 정책(인간 입력/AI 정책)
- 출력: 턴 로그, 행동 로그, 최종 점수, 성능 지표
- 이 모델을 UI와 분리해 테스트/리플레이/대회 집계에 재사용
