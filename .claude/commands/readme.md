# README.md 업데이트

프로젝트 전체 파일을 분석하여, **무엇을 시도했고 결과가 어땠는지** 중심으로 README.md를 다시 작성합니다.

## 작성 절차

### 1. 프로젝트 전체 파일 읽기

아래 영역을 빠짐없이 탐색합니다:

- `src/core/` — 게임 코어 로직
- `src/llm/` — 브라우저 LLM 자동 플레이, 메모리
- `src/` 나머지 — UI, 프로필, 랭킹, 도감 등
- `sandbox/sim/` — 헤드리스 시뮬레이션, 에이전트
- `docs/` — 보조 문서, devlog
- `data/` 또는 `src/asset/` — 카드 데이터, CSV 등
- `package.json`, `server.js` 등 — 인프라/서버 구성
- 루트의 설정 파일들

각 파일의 역할과 주요 구현 내용을 파악합니다.

### 2. "시도 → 결과" 분석

파일을 읽으면서 다음을 추출합니다:

- **어떤 기능/시스템을 시도했는가** (예: LLM 자동 플레이, 시장 이벤트 시스템, 헤드리스 시뮬레이터 등)
- **어떻게 구현했는가** (핵심 파일, 접근 방식)
- **실제로 작동하는 수준은 어디까지인가** (완성도, 동작 범위)
- **현재 한계나 남은 과제는 무엇인가**

단순히 "이런 파일이 있다"가 아니라, **"이걸 시도해서, 여기까지 됐고, 이건 아직이다"** 관점으로 정리합니다.

### 3. README.md 작성

아래 구조로 README.md를 작성합니다. 섹션 순서와 이름은 프로젝트 상황에 맞게 조정할 수 있습니다.

```markdown
# 프로젝트 이름

한두 줄로 프로젝트가 뭔지 설명.

> 비상업적/학습/프로토타입 고지 (필요시)

## 한 줄 요약

이 프로젝트의 현재 가치를 한 문장으로.

## 시도한 것과 결과

이 섹션이 README의 핵심입니다. 각 시도를 아래 형식으로 정리합니다:

### [시도 이름] (예: LLM 브라우저 자동 플레이)

- **뭘 하려 했나**: 목표/동기
- **어떻게 만들었나**: 핵심 파일과 접근 방식 (파일 경로 포함)
- **어디까지 됐나**: 실제 작동 범위, 성과
- **아직 남은 것**: 한계, 미완성 부분

(시도마다 반복)

## 게임 플레이 범위

현재 코드 기준으로 실제 플레이 가능한 범위를 간결하게 나열합니다.

## 실행 방법

```bash
npm install
npm run dev
```

핵심 명령어만 간결하게.

## 폴더 가이드

주요 디렉토리와 역할을 한 줄씩.
```

## 작성 원칙

- **한국어**로 작성합니다.
- **"시도 → 결과" 중심**: 독자가 "아, 이런 걸 해봤고 결과는 이랬구나"를 빠르게 파악할 수 있게 합니다.
- **정직하게**: 잘 된 것과 안 된 것을 둘 다 씁니다. 과장하지 않습니다.
- **파일 경로 포함**: 각 시도의 핵심 파일을 명시하여, 코드를 직접 보고 싶은 사람이 바로 찾아갈 수 있게 합니다.
- **간결하게**: 불필요한 서론이나 반복을 피합니다.
- **현재 코드 기준**: 초기 기획이 아니라, 지금 실제로 있는 코드를 기준으로 씁니다.
- 이모지는 사용하지 않습니다 (깔끔한 기술 문서 톤).
